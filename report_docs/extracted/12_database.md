# 💾 Database Schema and Design

## Firestore Collections Overview

```
Database: Firestore (Cloud Hosted, Document-based)
Indexing: Automatic basic, custom composite indexes for complex queries
Security: Firebase Auth + Firestore Security Rules
Regions: us-central1 (configurable)
```

## Collection Specifications

### 1. journal_entries
**Purpose**: Core collection storing user journal entries

```
Collection Path: /journal_entries/{document_id}

Document Structure:
{
  "uid": string (indexed),           # Firebase Auth UID
  "entry_text": string,              # Full journal entry (up to 5000 chars)
  "title": string (optional),        # User-provided or auto-generated title
  "created_at": timestamp (indexed), # Entry creation time
  "updated_at": timestamp            # Last modification time
}

Indexes:
  - uid, created_at (compound index) - For listing entries by user + date
  - uid alone - For filtering by user
  - created_at alone - For timeline queries

Query Examples:
  - Get all entries for user: WHERE uid == "user123" ORDER BY created_at DESC
  - Get entries in date range: WHERE uid == "user123" AND created_at >= date1 AND created_at <= date2
  - Get single entry: WHERE document_id = "entry456"

Estimated Size (per entry): 500 bytes average + entry_text
Document Limit: No hard limit, internal testing to 100k entries per user
```

### 2. entry_analysis
**Purpose**: Stores AI-generated analysis (mood, summary) for each entry

```
Collection Path: /entry_analysis/{document_id}

Document Structure:
{
  "entry_id": string (ref),         # Reference to journal_entries document ID
  "mood": object,                    # Mood probabilities from RoBERTa
  {
    "anger": number (0-1),
    "disgust": number (0-1),
    "fear": number (0-1),
    "happy": number (0-1),
    "neutral": number (0-1),
    "sad": number (0-1),
    "surprise": number (0-1)
  },
  "summary": string,                # Generated by BART (20-128 tokens)
  "emotional_state": object (optional),    # Interpreted emotions
  "semantic_context": string (optional),   # What the entry is about
  "temporal_context": string (optional),   # When/how this fits in user's timeline
  "recommendation_strategy": string (optional), # How recos should adapt
  "raw_analysis": object (optional),  # Full raw ML output
  "created_at": timestamp           # When analysis was created
}

Indexes:
  - entry_id - For looking up analysis by entry
  - created_at - For timeline queries

Query Examples:
  - Get analysis for entry: WHERE entry_id == "entry456"
  - Get analyses by mood: WHERE mood.happy > 0.7 (post-filtering in client)

Separation Rationale:
  - Separates entry content from analysis
  - Allows re-analysis without touching entry
  - Different access patterns (read entry often, read analysis sometimes)
```

### 3. journal_embeddings
**Purpose**: Stores semantic embeddings for similarity-based search

```
Collection Path: /journal_embeddings/{document_id}

Document Structure:
{
  "uid": string (indexed),            # User ID
  "entry_id": string,                # Reference to journal entry
  "embedding": array<number>,        # 384-dimensional embedding (float32)
  "created_at": timestamp (indexed) # When embedding was generated
}

Indexes:
  - uid, created_at - For fetching recent embeddings
  - entry_id - For looking up embedding for specific entry

Size Per Document: ~1.5 KB (384 floats per embedding)
Total Size for 10k entries: ~15 MB per user

Query Examples:
  - Get recent embedding for a user: WHERE uid == "user123" ORDER BY created_at DESC LIMIT 1
  - Get all embeddings in date range: WHERE uid == "user123" AND created_at BETWEEN dates

Future Use:
  - Vector search: Find similar entries (not yet implemented via API)
  - Semantic highlighting: Show related passages
```

### 4. user_vectors
**Purpose**: Stores taste preference vectors updated via interaction feedback

```
Collection Path: /user_vectors/{uid}

Document Structure:
{
  "uid": string (doc ID = uid),     # User ID as document ID
  "movies_vector": array<number>,   # 384D taste vector for movies
  "songs_vector": array<number>,    # 384D taste vector for songs
  "books_vector": array<number>,    # 384D taste vector for books
  "podcasts_vector": array<number>, # 384D taste vector for podcasts
  "updated_at": timestamp           # Last update time
}

Initialization:
  - Created on first recommendation request
  - Initialized to random normalized vector
  - All dimensions set equal (uniform preference)

Update Mechanism:
  - Updated when user provides feedback (click, save, skip)
  - Used in intent vector blending: 95% taste + 5% journal
  - Updated with interaction signal weights:
    - click: +0.02
    - save: +0.05
    - skip: -0.01
    
  Vector Update Formula:
    new_vector = (old_vector * weight + item_embedding * signal_weight) / ||...||

Query Examples:
  - Get user taste: WHERE uid == "user123"
  - Update taste: SET movies_vector = new_value

Persistence:
  - Survives across sessions
  - Accumulates feedback over time
  - Can be reset by admin if needed
```

### 5. insights
**Purpose**: Stores LLM-generated insights from journal analysis

```
Collection Path: /insights/{document_id}

Document Structure:
{
  "uid": string (indexed),              # User ID
  "start_date": string,                # YYYY-MM-DD format
  "end_date": string,                  # YYYY-MM-DD format
  "goals": array<object>,              # [{"title": "...", "description": "..."}]
  "progress": string,                  # Short progress description
  "negative_behaviors": string,        # Identified patterns to work on
  "remedies": string,                  # Suggested improvements
  "appreciation": string,              # Positive things to celebrate
  "conflicts": string,                 # Identified tensions/conflicts
  "raw_response": string,              # Full LLM response (for debugging)
  "created_at": timestamp (indexed)    # When insights were generated
}

Indexes:
  - uid, created_at - For listing user's insights chronologically

Query Examples:
  - Get all insights for user: WHERE uid == "user123" ORDER BY created_at DESC
  - Get insights in date range: WHERE uid == "user123" AND created_at BETWEEN dates

Typical Document Size: 2-5 KB
```

### 6. insight_entry_mapping
**Purpose**: Tracks which entries contributed to each insight

```
Collection Path: /insight_entry_mapping/{document_id}

Document Structure (Simple):
{
  "insight_id": string,   # Reference to insights document
  "entry_id": string      # Reference to journal_entries document
}

Purpose:
  - Allows reverse lookup: Given insight, find contributing entries
  - Allows cascade deletion: Delete insight → delete all mappings
  - Allows impact analysis: How many entries informed the insight?

Query Examples:
  - Get entries for insight: WHERE insight_id == "insight123"
  - Get insights influenced by entry: WHERE entry_id == "entry456"
  - Cascade delete mappings: DELETE WHERE insight_id == "insight123"

Document Count:
  - One per entry per insight
  - If insight covers 7 days with 10 entries = 10 mappings
  - Small documents, efficient storage
```

### 7. users
**Purpose**: User profile and settings

```
Collection Path: /users/{uid}

Document Structure:
{
  "uid": string (doc ID),          # User ID as document ID
  "email": string,                 # Email address
  "settings": object,              # User preferences
  {
    "mood_tracking_enabled": boolean,  # Enable/disable mood detection
    "language": string (default "en"),
    "timezone": string (default "Asia/Kolkata"),
    "theme": string (default "light")
  },
  "profile": object,               # Optional profile information
  {
    "display_name": string,
    "avatar_url": string,
    "bio": string
  },
  "created_at": timestamp,         # Account creation date
  "updated_at": timestamp
}

Auto-Create:
  - Created on first journal entry
  - Settings initialized with defaults
  - No manual registration needed (Firebase handles auth)

Query Examples:
  - Get user settings: WHERE uid == "user123"
  - Update settings: SET settings.mood_tracking_enabled = false
```

### 8. user_interactions
**Purpose**: Tracks user feedback on recommended media (for learning)

```
Collection Path: /user_interactions/{document_id}

Document Structure:
{
  "uid": string (indexed),          # User ID
  "media_type": string (indexed),   # "movies" | "songs" | "books" | "podcasts"
  "media_id": string,               # External media ID (TMDb, Spotify, etc.)
  "signal": string,                 # "click" | "save" | "skip"
  "context": string,                # "recommendation" | "search"
  "timestamp": timestamp (indexed)  # When interaction occurred
}

Indexes:
  - uid, media_type, timestamp - For rate limiting and feedback aggregation
  - uid, timestamp - For user activity timeline

Query Examples:
  - Rate limit check: WHERE uid == "user123" AND media_type == "movies" AND timestamp >= last_hour
  - Build feedback history: WHERE uid == "user123" ORDER BY timestamp DESC
  - Aggregate feedback: COUNT WHERE media_id == "tmdb_550" AND signal == "save"

Signal Weights (for taste vector updates):
  - click: +0.02
  - save: +0.05
  - skip: -0.01

Rate Limiting:
  - Max 10 interactions per media type per hour
  - Check: COUNT(WHERE uid == user AND media_type == type AND timestamp >= now - 1hour) < 10

Document Growth:
  - ~100-200 interactions per user per month
  - Old interactions (>90 days) can be archived
```

### 9-12. Media Cache Collections

**Purpose**: Cache media data from external providers (TMDb, Spotify, Google Books)

#### 9. media_cache_movies
```
Collection Path: /media_cache_movies/{document_id}

Document Structure:
{
  "tmdb_id": string,                # TMDb API ID
  "title": string,
  "description": string,
  "genre": array<string>,           # ["Drama", "Thriller"]
  "rating": number (0-10),
  "popularity": number (0-100),
  "release_date": string (YYYY-MM-DD),
  "image_url": string,              # Poster image
  "mood_tags": array<string>,       # [" happy", "inspiring"] for mood filtering
  "embedding": array<number>,       # 384D embedding of title+description
  "last_updated": timestamp,        # When cache entry was refreshed
  "validity_hours": number          # How long to cache (default 24)
}

Cache Refresh:
  - Fetch from TMDb API periodically
  - Update every 24 hours
  - Keep top 300-500 movies by popularity
  - Size per entry: ~1-2 KB

Query Examples:
  - Get all movies: No WHERE clause (fetch all for ranking)
  - Filter by genre: WHERE genre CONTAINS "Drama" (post-query in service)
  - Find by title: Fuzzy match on title (in-app, not Firestore)
```

## Data Relationships Diagram

```
journal_entries (core)
├── 1──→ entry_analysis (mood + summary)
│   (one analysis per entry)
│
├── 1──→ journal_embeddings (semantic vector)
│   (one embedding per entry)
│
├── 1──→ insight_entry_mapping ──→ insights
│   (many-to-many: entry → inquiry contributing insights)
│
└── → user_vectors (blended into intent)
    (used for recommendations)

user_vectors
├── → recommendation_pipeline (intent = 95% taste + 5% journal)
│
└── ← user_interactions (updated by feedback)
    (click, save, skip signals)

media_cache_*
├── → recommendation_pipeline (candidates for ranking)
│
└── ← user_interactions (tracked for feedback)
    (learning which media users prefer)

users
├── → journal_entries (every entry has owner uid)
├── → entry_analysis (indirect via entry)
├── → insights (every insight has owner uid)
└── → user_interactions (tracks user feedback)
```

## Firestore Rules (Security)

```
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    
    // Journal entries: read/write only own entries
    match /journal_entries/{document=**} {
      allow read, write, delete: if request.auth != null && 
                                   request.resource.data.uid == request.auth.uid;
      allow create: if request.auth != null && 
                       request.resource.data.uid == request.auth.uid;
    }
    
    // Entry analysis: read own, write public (service)
    match /entry_analysis/{document=**} {
      allow read: if request.auth != null;
      allow write: if request.auth != null;  // Typically written by backend service
    }
    
    // Journal embeddings: read own, write backend
    match /journal_embeddings/{document=**} {
      allow read: if request.auth != null && 
                     request.resource.data.uid == request.auth.uid;
      allow write: if request.auth != null;  // Backend service
    }
    
    // User vectors: read own, write backend
    match /user_vectors/{uid} {
      allow read: if request.auth != null && request.auth.uid == uid;
      allow write: if request.auth != null;
    }
    
    // Insights: read/write own insights
    match /insights/{document=**} {
      allow read, write, delete: if request.auth != null && 
                                   request.resource.data.uid == request.auth.uid;
    }
    
    // Insight-entry mappings: backend only
    match /insight_entry_mapping/{document=**} {
      allow read: if request.auth != null;
      allow write: if request.auth != null;
    }
    
    // User settings: read/write own
    match /users/{uid} {
      allow read, write: if request.auth != null && request.auth.uid == uid;
    }
    
    // Interactions: read/write own, rate limit enforced in code
    match /user_interactions/{document=**} {
      allow read, write: if request.auth != null;
    }
    
    // Media caches: read public
    match /media_cache_{type}/{document=**} {
      allow read: if request.auth != null;
      allow write: if request.auth != null;  // Backend only (service account)
    }
  }
}
```

## Design Decisions

| Decision | Rationale | Trade-off |
|----------|-----------|-----------|
| **Separate entry_analysis** | Different access patterns, allows re-analysis | Extra document, join query |
| **Separate embeddings collection** | Large arrays (384 floats), not needed for all queries | Additional storage |
| **user_vectors in DB** | Fast recommendations without recomputation | Need continuous updates |
| **Compound indexes** | Fast list queries with date filtering | Index maintenance cost |
| **Denormalized mood in analysis** | Direct mood access without entry fetch | Data duplication |
| **Media cache collections** | ~300-500 items per type, easy filtering | Cache refresh overhead |

## Storage Estimates

```
Per User (after 1 year active use):

journal_entries: 365 entries × 0.5 KB = 183 KB
entry_analysis: 365 × 1.5 KB = 548 KB
journal_embeddings: 365 × 1.5 KB = 548 KB
insights: 50 insights × 2 KB = 100 KB
insight_entry_mapping: 365 mappings × 0.1 KB = 37 KB
users: 1 × 0.5 KB = 0.5 KB
user_interactions: 365 × 0.2 KB = 73 KB
user_vectors: 1 × 1.5 KB = 1.5 KB

TOTAL PER USER: ~1.5 MB

For 10,000 users: ~15 GB (well within Firestore free tier for reads/writes)
```

## Scaling Considerations

- **Document Limit**: Firestore supports >1B documents, no hard limit per collection
- **Query Speed**: Composite indexes ensure <100ms queries at scale
- **Write Capacity**: Firestore auto-scales, no manual provisioning
- **Archival**: Move old interactions (>90 days) to archive collection for cost savings
