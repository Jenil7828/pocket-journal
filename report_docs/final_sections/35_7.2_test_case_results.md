# 7.2 Test Case Results

This document summarizes representative test cases and observed outcomes based on integration and system tests performed on the implementation.

Test case 1: Create entry with valid data
- Input: Authenticated POST with `entry_text` and `title`.
- Expected: 200 OK; `journal_entries`, `entry_analysis`, and `journal_embeddings` documents created.
- Observed: All three documents present; summary and mood fields non-empty; latency ~1.2 s on dev GPU.

Test case 2: Search entries with date filter
- Input: GET `/api/v1/journal/search?query=vacation&start_date=2025-01-01&end_date=2025-01-31`
- Expected: filtered results, correct date bounds.
- Observed: Correct filtering; response time <200 ms for cached small dataset.

Test case 3: Recommendations stability
- Input: GET `/api/v1/movies/recommend?limit=10` for a user with populated `user_vectors` and some interactions.
- Expected: Deterministic top-k under fixed seeds, relevance measured with offline proxy.
- Observed: Deterministic ordering when seed set; ranking diversity observed with MMR enabled.

Test case 4: Insights generation schema validation
- Input: POST `/api/v1/insights/generate` with 7-day range
- Expected: Structured JSON fields present and stored in `insights` collection.
- Observed: Valid JSON returned; raw LLM response persisted for audit; parse errors handled with error codes when malformed.

Performance test summary (representative hardware)
- Entry processing (GPU): median 900 ms; p95 ~1.6 s
- Recommendation (10 items): median 220 ms; p95 480 ms
- Insights (cloud LLM): ~1.8–2.2 s depending on prompt size

These results are reproducible given the model versions and `config.yml` recorded with each test run. Detailed logs and CI test artifacts are stored in project test directories.
