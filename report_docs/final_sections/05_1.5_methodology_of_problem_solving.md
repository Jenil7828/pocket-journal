# 1.5 Methodology of Problem Solving — Four Balanced Pipelines

This section presents four equally emphasised pipelines that implement the system's functionality: (1) Journaling Pipeline, (2) Analysis Pipeline, (3) Recommendation Pipeline and (4) Insight Generation Pipeline. Each pipeline is specified in terms of inputs, transformations, outputs, justification and edge-case handling.

1. Journaling Pipeline

1.1 Purpose
- Ingest and persist user-authored journal submissions as authoritative records suitable for downstream analysis and auditing.

1.2 Input
- User-authored journal text (mandatory) and optional title; submissions are accepted within an authenticated user session. Operational configuration caps text length to control storage and downstream costs.

1.3 Steps and transformations
- 1.3.1 Ingestion: accept submission, record arrival timestamp and source metadata for provenance.
- 1.3.2 Validation: perform presence and type checks and enforce length constraints; reject or truncate according to policy; return descriptive diagnostics on failure.
- 1.3.3 Canonical persistence: write validated record to durable storage, assign a stable identifier and record creation/update timestamps; establish indices for time-range and user-scoped queries.
- 1.3.4 Instrumentation: emit metrics for ingestion latency, write success/failure and queue analytic tasks; persist provenance data to support reproducibility.

1.4 Output
- A canonical persisted entry with provenance metadata and indexing sufficient for efficient retrieval and reproducible downstream processing.

1.5 Justification
- Reliable ingestion and canonical persistence ensure that analytic artifacts have an authoritative origin. Without them, aggregation, traceability and auditability are compromised.

1.6 Edge cases
- Partial writes or indexing failures are logged and flagged; persisted raw submissions can be retained pending index repair. Systems handling delayed analytic tasks will proceed using available canonical data.

2. Analysis Pipeline

2.1 Purpose
- Convert canonical text into machine-actionable artifacts: concise summary, calibrated affective probabilities and compact semantic embeddings. These artifacts are the primary inputs to recommendation and insight pipelines.

2.2 Input
- Canonical entry text and ingestion provenance metadata (timestamps, source identifiers).

2.3 Steps and transformations
- 2.3.1 Normalization: trim, normalize whitespace and remove non-printable control characters to stabilise inputs.
- 2.3.2 Summarization: generate an abstractive summary under constrained generation parameters to produce a concise human-readable surrogate; persist generation metadata.
- 2.3.3 Affective quantification: compute calibrated probabilities across implemented emotion dimensions; persist threshold and confidence indicators alongside probabilities.
- 2.3.4 Embedding: produce a fixed-length numeric vector (dimension 384) from the summary and L2-normalize to unit length for cosine-similarity operations; persist vector and provenance.

2.4 Output
- Per-entry artifacts: (a) summary text and generation metadata, (b) per-dimension probability vector and threshold metadata, (c) normalized embedding vector with timestamp.

2.5 Justification
- The trio of summary, affect probabilities and embedding provides complementary representations: human-readable condensation, graded affective signal and geometry-preserving numeric encoding for efficient similarity computations.

2.6 Constraints and impact
- Generation and tokenisation limits bound inference cost and control latency; truncation can omit distant context and reduce sensitivity for long narratives. Embedding dimensionality is a trade-off between representational fidelity and storage/compute costs.

2.7 Edge cases
- Deterministic fallback behaviour is applied when generation fails (persisted truncated prefix for summary); missing embeddings are recorded as absent and treated as neutral in numeric computations to prevent propagation of failures.

3. Recommendation Pipeline

3.1 Purpose
- Produce ranked, diverse media recommendations that reflect both long-term preference and immediate contextual signals derived from recent journaling.

3.2 Input
- User identifier, the most recent entry-derived embedding when available, persistent preference representations, and optional selectors (genre, mood label, free-text query, language) plus pagination parameters.

3.3 Steps and transformations
- 3.3.1 Intent formation: blend persistent preference vectors with recent entry embeddings using conservative weighting to form a normalized intent vector that preserves historical taste while incorporating short-term context.
- 3.3.2 Candidate retrieval: obtain a bounded candidate pool from maintained caches filtered by media type and explicit criteria.
- 3.3.3 Base scoring: compute cosine similarity between intent and candidate embeddings as the primary relevance measure.
- 3.3.4 Hybrid ranking: combine similarity with interaction frequency, item popularity and recency using calibrated weights; apply an exponential temporal decay factor to older interactions to prioritise recent signals.
- 3.3.5 Diversity step: apply Maximal Marginal Relevance (MMR) with tunable λ to select a diverse and relevant set, where
  MMR_score(c) = λ × relevance(c) − (1 − λ) × max_{s in selected} similarity(c, s).
- 3.3.6 Finalisation: apply requested sort order, paginate and redact internal scoring artifacts before returning item metadata.

3.4 Output
- A paginated list of recommended items (redacted metadata) with pagination information and explicit flags for fallback or defaulted behaviour (e.g., cold-start cases).

3.5 Justification
- Blending short- and long-term signals produces stable yet responsive recommendations; MMR enhances variety and reduces redundancy, improving user discovery and engagement.

3.6 Edge cases
- Cold-start users without historical vectors receive default popular-item lists; constrained candidate pools trigger filter relaxation strategies with explicit client flags.

4. Insight Generation Pipeline

4.1 Purpose
- Synthesize multi-entry interpretations over specified windows to produce validated structured insight documents that support user reflection and research analysis.

4.2 Input
- Date window (start and end dates) and persisted per-entry artifacts (summaries, affective probabilities and embeddings) within that window.

4.3 Steps and transformations
- 4.3.1 Aggregation: compute summary statistics (entry count, per-dimension mean probabilities, frequency of high-probability detections) to characterise the window.
- 4.3.2 Context assembly: select up to N per-entry summaries by recency and computed salience to form a bounded context for synthesis and to respect generation capacity limits.
- 4.3.3 Structured synthesis: request a constrained-format interpretation that yields required fields (goals, progress, recurring behaviours, remedies, appreciations, conflicts) and return a candidate structured document.
- 4.3.4 Validation and fallback: perform schema validation of the candidate; where malformed, attempt schema-aligned extraction from the raw generator output; persist raw output and flag instances requiring human review.
- 4.3.5 Persistence and mapping: persist validated insights and map contributing entries to enable traceability and evidence-based inspection.

4.4 Output
- A validated structured insight document with required interpretive fields and provenance; metadata includes contributing entry count and generation timestamp.

4.5 Justification
- Structured insights provide actionable, auditable interpretations of longitudinal patterns; they convert distributed per-entry signals into coherent narratives usable for intervention and research.

4.6 Edge cases
- When assembled context exceeds generation capacity, the pipeline prioritises most recent and highest-salience entries; partially completed schemas are recorded with placeholders and flagged for human review.

COMMON VALIDATION, MONITORING AND FAILURE HANDLING

- All produced artifacts include provenance metadata (generation parameters, timestamps and identifiers) to enable reproducibility and controlled experiments.
- Failures in optional analytic components do not prevent canonical storage; artifacts are marked as absent and deterministic fallbacks are applied where appropriate.
- Monitoring tracks ingestion throughput, analytic latencies and error rates; under constrained resources, heavy workloads are scheduled asynchronously to preserve responsiveness for interactive operations.

This methodology articulates four balanced pipelines with explicit inputs, transformations, outputs, justification and robust failure handling to ensure the system coherently supports journaling, analysis, recommendation and insight generation.
