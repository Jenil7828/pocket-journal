# 1.4 Scope and Limitations

This section enumerates precisely which inputs, outputs and processing behaviors the system supports, and documents limitations that are directly observable from the implemented system and configuration.

1. Supported inputs
   - Text-based journal entries only. Each accepted entry must include the full journal text (UTF-8 string, non-empty) and may include an optional short title (UTF-8 string, recommended maximum about 200 characters). The system enforces a practical maximum on journal text length (around 5000 characters) via configuration.
   - Insight requests are specified by a date window expressed in standard calendar date format (start and end dates in YYYY-MM-DD form).
    - Recommendation requests are issued for a particular user and may include optional selection criteria such as genre, mood label, free-text search terms and language; responses are returned in paginated form according to requested limits and offsets.

    - Justification:
      - Why inputs are constrained: limiting inputs to structured date windows and text entries reduces ambiguity in aggregation and simplifies validation and provenance tracking.
      - Operational impact: clients must conform to textual entry formats and date-window semantics when requesting insights and recommendations.
2. Supported outputs

    - Per-entry analytic record including:
        - Summary (string)
        - An emotion probability mapping over seven discrete labels, each mapped to a floating-point probability in [0,1].
        - A semantic embedding vector of fixed dimension (384) represented as a list of floating-point values that is L2-normalized for similarity computations.
      - Recommendation responses: paginated lists of media items (metadata only) with pagination metadata and applied filter echo.
      - Insight documents: validated structured documents with required interpretive fields (goals, progress, negative_behaviors, remedies, appreciation, conflicts) and metadata (entry count, generation timestamp).

    - Justification:
      - Why these outputs are provided: they form a compact set of machine-actionable artifacts enabling retrieval, personalization and longitudinal synthesis while minimizing exposure of raw text for downstream use.
      - Operational impact: downstream analyses and external exports operate primarily on these standardized artifacts rather than raw narrative text.

3. Processing included (per-entry)
   - Validation of input fields and length constraints.
   - Persistence of canonical entry record with creation and update timestamps.

      - Computation of emotion probabilities from tokenized text subject to an effective token limit for this stage (approximately 128 tokens).
         - Abstractive summary generation for inputs up to practical generation limits (observed around 1024 tokens), producing concise summaries typically within the 20–128 token range when generation proceeds.
         - Generation of a normalized 384-dimensional embedding from the summary, used for cosine-similarity computations.
         - Incremental blending of the new embedding into persistent user preference vectors where historical preference receives dominant weight and the new embedding provides a smaller update contribution.

      - Justification:
        - Why processing stages are sequenced: summary generation reduces input dimensionality and focuses semantic content; embedding converts that content into a geometry-preserving numeric form; blending updates personalization incrementally, avoiding abrupt shifts.
        - Impact on system behavior: these stages improve retrieval precision, maintain personalization stability and support efficient nearest-neighbour computations for recommendations.

4. Explicit limitations (derived from implementation)
   - Input modality: The system does not accept audio, image or other non-textual inputs. Multimodal processing is not supported.
    - Tokenization constraints: Emotion analysis operates on tokenized input truncated or padded to 128 tokens; summaries are generated from text truncated to approximately 1024 tokens. Inputs exceeding these limits will be truncated according to configuration.
   - Embedding dimensionality: All embeddings are 384-dimensional and normalized; alternative embedding dimensions are not supported by the current deployment.
   - Dependency on pretrained inference artifacts: The analytic capabilities depend on externally provisioned pretrained models loaded at run-time; the presence and performance of these models is required for analysis (the system provides fallbacks when models are unavailable, but full analytic output will be incomplete).
   - Latency considerations: Target latency ranges for inference are documented as design objectives (e.g., emotion inference and embedding generation operate in sub-second ranges on accelerated hardware); actual latency depends on available hardware and may be higher on CPU-only deployments.
   - Cold-start personalization: For users lacking prior interaction data, recommendations default to cached popular items and default preference vectors; recommendation quality is expected to improve as user interactions accumulate.

5. Failures and degradation modes (observed behaviors)
   - If the emotion analysis component is unavailable the system persists the entry and returns an explicit indication that mood probabilities are absent.
   - If the summarization component fails, a truncated version of the entry (first ~200 characters) is used as a fallback summary and the failure is logged.
   - If embedding generation fails, the downstream recommendation pipeline treats the missing embedding as a zero contribution for intent blending.

  - Edge-case explanations:
    - Why failures occur: resource exhaustion, transient model or service unavailability, and malformed inputs can trigger analytic component failures.
    - How the system handles them: canonical storage is preserved; analytic artifacts are marked absent or replaced with deterministic fallbacks; monitoring and logging capture diagnostic information for remediation.

6. Not explicitly supported by the system
   - Real-time collaborative multi-user journaling (simultaneous shared editing) is not provided.
   - Clinical-grade diagnostic outputs are not produced; insights are for user reflection and research and are not validated clinical instruments.

These scope and limitation statements are derived from the system's configured input constraints, observed data artifacts and documented fallback behaviors; no speculative features are included.
