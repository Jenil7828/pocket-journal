# 3.1 Assumptions and Dependencies

## Assumptions
1. Authenticated environment: The system assumes clients authenticate using Firebase ID tokens; all endpoints requiring user context rely on a valid `uid` claim.
2. Model availability: Pretrained model artifacts (RoBERTa, BART, all-mpnet-base-v2, Qwen2) are available on local disk or accessible via configured model store paths.
3. Network reliability for providers: External provider APIs (TMDb, Spotify, Google Books) are reachable; the system can fall back to cached data when provider calls fail.
4. Firestore availability: The Firestore instance is available and supports the configured index queries.

## External Dependencies
- Firebase Admin SDK (Auth, Firestore)
- PyTorch and HuggingFace Transformers for model inference
- sentence-transformers library for embedding generation
- External media provider APIs: TMDb, Spotify, Google Books
- Optional: Ollama or local model runtime for Qwen2 inference

## Configuration Dependencies
- `config.yml` contains critical runtime flags (enable_llm, model versions, recommendation parameters) and must be present at startup.
- Environment variables: MODEL_STORE_PATH, FIREBASE_SERVICE_ACCOUNT, and provider API keys (TMDB_API_KEY, SPOTIFY_CLIENT_ID/SECRET) when live provider access is required.

## Operational Assumptions
- GPU availability improves latency but CPU fallback is supported; float16 precision is used on CUDA where configured.
- Rate limits for provider APIs are enforced by caching strategies and retry/backoff policies in `services/media_recommender/providers`.

These assumptions and dependencies define preconditions for deployment and experimental reproducibility.
