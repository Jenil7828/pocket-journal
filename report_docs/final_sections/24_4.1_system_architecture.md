# 4.1 System Architecture (Layered Design)

The system employs a layered architecture that isolates concerns and enables empirical evaluation. Layers and responsibilities:

1. Presentation Layer (API)
   - Flask-based REST endpoints located in `routes/` (journal_domain, media_domain, insights_domain).
   - Responsibilities: request validation, authentication (`@login_required`), and response formatting.

2. Service Layer (Business Logic)
   - Domain services located in `services/` (journal_entries, media_recommender, insights_service).
   - Responsibilities: orchestrate ML inference, database transactions, ranking logic and caching policies.

3. Inference Layer (ML)
   - Predictor wrappers for RoBERTa (mood detection), BART (summarization), EmbeddingService (all-mpnet-base-v2) and LLM-based insights (Gemini/Qwen2) in `ml/inference/`.
   - Responsibilities: encapsulate model loading, device selection, batching and output normalization.

4. Persistence Layer (Data Storage)
   - Firestore collections and `persistence/db_manager.py` for CRUD operations and schema validation.
   - Data artifacts: `journal_entries`, `entry_analysis`, `journal_embeddings`, `user_vectors`, `insights`, `media_cache_*`.

5. External Services
   - Media providers (TMDb, Spotify, Google Books) and optional cloud LLM endpoints.
   - Provider wrappers implement caching and fallback to maintain latency and cost targets.

Design rationale:
- The layered approach improves testability and allows performance isolation: model inference concerns are confined to the inference layer, enabling benchmarking independent of API overhead.
- The document-oriented persistence model supports per-user isolation and time-bounded queries essential for longitudinal analysis.

Deployment considerations:
- Stateless Flask instances behind a load balancer for horizontal scaling.
- Local model caching per instance to avoid repeated downloads.
- Optional GPU-backed instances for inference-heavy workloads.

This architecture is instantiated in the repository and described in `09_architecture.md` and `12_modules.md` for traceability between design and implementation.
