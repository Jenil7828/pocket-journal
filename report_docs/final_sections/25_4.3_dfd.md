# 4.3 Data Flow Diagram (DFD)

This section describes the logical data flows within the system using the implemented pipelines as the basis for the diagrammatic model. The DFD components are described in prose to map to the codebase.

Entities:
- User (actor): Submits entries and requests recommendations/insights.
- External Providers: TMDb, Spotify, Google Books; used for media metadata.

Processes and flows:
1. Entry ingestion:
   - Flow: User → `POST /api/v1/journal` → `journal_entries` → inference tasks → `entry_analysis` & `journal_embeddings`.
   - Data stores: `journal_entries`, `entry_analysis`, `journal_embeddings`.

2. Recommendation request:
   - Flow: User → `/api/v1/{media}/recommend` → intent builder (fetch recent embeddings + user_vectors) → candidate retrieval from `media_cache_*` → ranking engine → response to user.
   - Data stores: `user_vectors`, `media_cache_*`, `user_interactions`.

3. Insights generation:
   - Flow: User → `/api/v1/insights/generate` → fetch entries + analysis → LLM prompt → LLM response → `insights` & `insight_entry_mapping`.
   - Data stores: `insights`, `insight_entry_mapping`.

Data movement notes:
- Embeddings move from the inference layer to Firestore (`journal_embeddings`) and are used by the recommendation pipeline.
- Interaction events written to `user_interactions` update `user_vectors` and influence subsequent recommendations.

This DFD mapping is directly implementable as a visual diagram using the process and data-store nodes described above; it is traceable to the implementation in `services/` and `persistence/` modules.
